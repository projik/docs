Network failures, model timeouts, and harness restarts all lead to retried requests — and retried POST requests without idempotency create duplicate records. dAvePi implements a Stripe-style idempotency contract on every auto-generated POST route and every `create_<path>` MCP tool. Tag each logical operation with a stable key, and the server guarantees it runs at most once, no matter how many times you retry.

## Pass the Idempotency-Key Header

Include an `Idempotency-Key` header on any POST request:

```http
POST /api/v1/account
Authorization: Bearer <token>
Idempotency-Key: 9f3c1c2e-4b5a-4d6e-8f3a-1b2c3d4e5f6a
Content-Type: application/json

{ "accountName": "Acme" }
```

The first call executes the operation and returns `201 Created`. Any retry with the **same key and same body** within 24 hours returns the original cached response — without executing the handler again — and adds a replay indicator header:

```http
HTTP/1.1 201 Created
Idempotency-Replay: true
Content-Type: application/json

{ "_id": "65b1...", "accountName": "Acme" }
```

## Key Scoping Rules

Keys are scoped per `(key, user, route)`:

- **Per user** — User A and User B can use the same key string without colliding.
- **Per route** — the same key on `POST /api/v1/account` and `POST /api/v1/contact` does not deduplicate across routes.
- **Per body** — the same `(key, user, route)` with a different request body returns a `409 IDEMPOTENCY_CONFLICT` error (see below).

## Conflict: Same Key, Different Body

If you reuse a key with a changed request body, the server rejects the request with `409`:

```json
{
  "error": {
    "code":    "IDEMPOTENCY_CONFLICT",
    "message": "Idempotency-Key was reused with a different request body"
  }
}
```

The body hash is computed over the effective post-filter payload (after ACL stripping and tenant stamping), and keys are sorted recursively before hashing. This means `{"a":1,"b":2}` and `{"b":2,"a":1}` produce the same hash and correctly replay rather than false-conflict.

<Warning>
Always use a **different** key for logically different operations. Reusing a key with a changed body signals an agent bug and is permanently rejected for the life of the original key's TTL.
</Warning>

## Non-2xx Responses Are Not Cached

If a request fails (validation error, 400, 500, etc.), the idempotency slot is released. The agent can fix its payload and retry under the **same key** — it will not be locked into the failed state.

<Note>
Only successful `2xx` responses are cached. A `400 VALIDATION` error clears the slot so you can correct the payload and retry with the same key.
</Note>

## Concurrent Retries

If two requests with the same key arrive simultaneously, the server uses an atomic claim-execute-complete protocol to ensure exactly one runs:

<Steps>
  <Step title="Claim">
    An atomic insert (backed by a unique MongoDB index on `{ key, userId, route }`) marks the slot `in_progress`. Exactly one concurrent caller wins.
  </Step>
  <Step title="Execute">
    The winner runs the handler. Losers inspect the existing row:
    - Same body, `in_progress` → `409 IDEMPOTENCY_IN_PROGRESS` ("retry shortly").
    - Same body, `completed` → replay the cached response.
    - Different body → `409 IDEMPOTENCY_CONFLICT`.
  </Step>
  <Step title="Complete or abandon">
    On a `2xx`, the winner promotes the slot to `completed` with the cached response. On a non-`2xx` or thrown error, the slot is deleted so the agent can retry under the same key with a corrected payload.
  </Step>
</Steps>

## Idempotency via MCP

MCP's JSON-RPC transport does not carry per-call HTTP headers, so pass the key as a tool argument instead:

```json
{
  "name": "create_account",
  "arguments": {
    "record": { "accountName": "Acme" },
    "idempotencyKey": "9f3c1c2e-4b5a-4d6e-8f3a-1b2c3d4e5f6a"
  }
}
```

Replays include `_idempotent_replay: true` on the result so your agent can distinguish a fresh create from a replayed one.

## Recommended Retry Pattern

Generate one UUID per logical operation and reuse it on every retry attempt for that operation:

```ts
import { randomUUID } from 'node:crypto';

async function safeCreate(payload: Record<string, unknown>) {
  const key = randomUUID(); // generate once per logical operation

  for (let attempt = 1; attempt <= 3; attempt++) {
    try {
      const response = await fetch('/api/v1/account', {
        method: 'POST',
        headers: {
          'Authorization':    `Bearer ${token}`,
          'Idempotency-Key':  key,     // same key on every retry
          'Content-Type':     'application/json',
        },
        body: JSON.stringify(payload),
      });
      return await response.json();
    } catch (transient) {
      if (attempt === 3) throw transient;
      // wait before retrying
      await new Promise((r) => setTimeout(r, 500 * attempt));
    }
  }
}
```

<Tip>
Use UUID v4 as your idempotency key. Any opaque string works, but UUIDs are collision-resistant and widely supported by API clients and automation frameworks.
</Tip>

## TTL and Expiry

Keys expire after **24 hours** by default. Override this with the `IDEMPOTENCY_TTL_SECONDS` environment variable:

```bash
IDEMPOTENCY_TTL_SECONDS=86400   # 24 hours (default)
```

MongoDB's TTL index sweeps expired records automatically on a background cycle. The framework also enforces an explicit `expiresAt > now()` filter on every lookup, so an expired row that the TTL monitor hasn't swept yet is treated as non-existent — expiry is a hard ceiling, not a best-effort window.

## Storage Shape

Idempotency records live in the `idempotency_key` collection:

```json
{
  "key":       "9f3c1c2e-...",
  "userId":    "65b1...",
  "route":     "POST /api/v1/account",
  "bodyHash":  "5c6f...",
  "state":     "completed",
  "status":    201,
  "body":      { "_id": "65b1...", "accountName": "Acme" },
  "headers":   { "Content-Type": "application/json" },
  "expiresAt": "2026-05-11T12:00:00Z",
  "createdAt": "2026-05-10T12:00:00Z"
}
```
