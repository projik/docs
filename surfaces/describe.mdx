`GET /_describe` returns a compact JSON manifest of everything dAvePi knows about your loaded schemas — fields, relations, aggregations, file fields, ACL slots, soft-delete and audit flags, REST endpoints, GraphQL queries and mutations, and MCP tool names. An agent or tool landing on a fresh dAvePi project can plan against the entire API in one round-trip, before writing a single line of integration code.

## Why not Swagger?

dAvePi also serves a Swagger 2.0 spec at `/api-docs/swagger.json`, but it has gaps for agent-first use:

| | Swagger 2.0 | `_describe` |
|-|-------------|-------------|
| Size | Large — full path matrix per resource | Compact — feature flags and per-schema metadata |
| Relations | Untyped (`accountId: string`) | First-class — `{ kind: "hasMany", target: "contact", fk: "accountId" }` |
| State machines | Untyped enum | First-class — `{ initial, states, transitions }` |
| Aggregations | Each is a path with hand-typed params | First-class — `[{ name, params }]` |
| ACL slots | Absent | Present — `{ list: ["admin"] }` |
| Idempotency support | Absent | Per-route flag |
| MCP tools | Absent | Per-schema list |

For an agent that needs to plan actions, `_describe` is the right map.

## How agents use it to plan

When an agent first connects to a dAvePi project, it issues one request:

```http
GET /_describe
```

From the response it learns:

- Every resource name and its `path` — which becomes the basis of every MCP tool name and REST route.
- Every field name, type, and flag (`required`, `searchable`, `computed`, `stamped`) — so it knows what to send and what the server will reject.
- Every relation — so it can decide whether to use `include` on a `get_<path>` call or issue a separate `list_<path>_<rel>` tool call.
- Every state machine — so it knows the valid transitions before attempting a `update_<path>` call that would be rejected.
- Every aggregation — so it can run analytics queries without guessing the tool name or parameter shape.
- Every MCP tool name — so it can advertise the correct tool list without calling `listTools()` first.

This planning step collapses what would otherwise be multiple exploratory API calls into a single round-trip.

---

## Auth

`/_describe` is **public by default** — it describes shape, not data. Knowing that an `account` resource exists does not expose any tenant's records; an agent still needs a valid JWT to call any generated tool or route.

If you want to gate it, wrap the route with the auth middleware in `app.js`:

```js
app.use('/_describe', auth(true), describeRouter);
```

---

## Stable contract

The `_describe` shape is part of dAvePi's stable API contract. The framework versions it via the `version` field and treats breaking changes the same as a major-version bump. New field flags can be added in minor releases without breaking consumers; the meaning of existing keys will not change.
