In development, dAvePi watches your `schema/versions/v1/` directory and reloads any changed schema file live. REST routes, GraphQL resolvers, Swagger fragments, and MCP tools all update within 50–150ms of a file save. Requests already in flight finish against the old surface; new requests hit the updated one immediately.

## Enable hot reload

Set the environment variable before starting the server:

```bash
HOT_RELOAD_SCHEMAS=true npm start
```

<Warning>
  Hot reload is gated on `NODE_ENV !== 'production' && HOT_RELOAD_SCHEMAS=true`. It never activates in production. Use [migrations](/operations/migrations) for schema changes on running production instances.
</Warning>

## What gets reloaded

Every file event triggers a precise, targeted rebuild — not a full server restart:

| Event | What rebuilds |
|-------|---------------|
| Add a new schema file | New Mongoose model, new REST router, new GraphQL type composer and resolvers, new Swagger fragment, new MCP tools. Connected MCP clients receive `tools/list_changed`. |
| Edit an existing schema | Mongoose model is `deleteModel`'d and re-registered, REST router replaced, GraphQL TC rebuilt, Swagger updated, MCP tools refreshed. |
| Delete a schema file | Router spliced from the Express stack, model deleted, GraphQL TC removed, Swagger fragment unset, MCP tools unregistered. |

## How it works

### Ordered rebuild queue

A file-watcher monitors `schema/versions/v1/` for `add`, `change`, and `unlink` events. Each event is queued and processed one at a time, so a rapid burst of saves (for example, when an agent edits several files at once) never causes partial or interleaved rebuilds.

### GraphQL: zero-downtime schema swap

Apollo Server builds its schema once at startup. To support live updates, dAvePi keeps a stable indirection layer in place from the moment the server boots. When a schema changes, the framework builds the new GraphQL schema in the background, swaps the live reference atomically, then shuts down the old server. Requests already in flight are served by the previous schema; new requests immediately hit the updated one — no dropped connections.

### REST: per-resource routing

Each resource's REST routes are isolated so only the changed resource is affected during a reload. When a schema is removed, its routes are cleanly torn down without touching any other resource in the stack.

### MCP: live tool registry

The stdio MCP server rebuilds its tool list on every schema change and notifies connected clients — Claude Desktop, Cursor, Claude Code — so they pick up added or removed tools without reconnecting.

The HTTP MCP transport at `POST /mcp` builds a fresh server per request, so hot-reload is automatic with no extra wiring.

## Performance

Hot reload runs entirely in-process — no fork, no restart. A typical single-schema reload completes in **50–150ms** on modest hardware (MongoDB index creation is the slow step). The single-flight queue coalesces a burst of saves: `rebuildGraphQL` only fires once the whole batch has settled.

## Correctness guarantees

The hot-reload cycle is covered by the framework's test suite:

- Adding and removing schemas brings their REST routes, Swagger fragments, and GraphQL fields up and down cleanly.
- Reloading the same schema is idempotent — no duplicate routes are mounted.
- Edits to an existing schema with new fields round-trip correctly on `POST` / `GET`.
- The watcher activates only when both `HOT_RELOAD_SCHEMAS=true` and `NODE_ENV` is not `production`.

## Development workflow

<Steps>
  <Step title="Start with hot reload enabled">
    ```bash
    HOT_RELOAD_SCHEMAS=true npm start
    ```
  </Step>
  <Step title="Edit a schema file">
    Add, change, or remove a field in any file under `schema/versions/v1/`. Save the file.
  </Step>
  <Step title="See the change live">
    Within 50–150ms the terminal logs the reload. Hit `/_describe`, Swagger, or your REST endpoint — the change is already live.
  </Step>
  <Step title="Verify in the manifest">
    ```bash
    curl -s http://localhost:5050/_describe \
      | jq '.schemas[] | select(.path=="deal").fields'
    ```
  </Step>
</Steps>
