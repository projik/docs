You run a small real estate brokerage. You want listings with photos, leads with a state machine that enforces the funnel, viewings linking them together, and your Slack to wake up when something interesting happens — without you writing any notification glue. The agent then answers analytics questions on demand. The key lesson in this tutorial is the difference between the *passive* plugin surface (one-way notifications) and the *active* agent surface (interactive queries and charts). Both run side by side in the same workspace.

## Prerequisites

- Completed the [Customer Support Inbox](/tutorials/customer-support-inbox) tutorial or equivalent familiarity with dAvePi schemas and the agent. The [common prerequisites](/tutorials/) apply.

## Write the three schemas

Create `schema/versions/v1/listing.js`:

```js
module.exports = {
  path: 'listing',
  collection: 'listing',
  fields: [
    { name: 'userId', type: String, required: true },
    { name: 'address', type: String, required: true, searchable: true, searchWeight: 5 },
    { name: 'neighborhood', type: String, searchable: true },
    { name: 'price', type: Number, required: true },
    { name: 'beds', type: Number, required: true },
    { name: 'baths', type: Number, required: true },
    { name: 'square_feet', type: Number },
    {
      name: 'status',
      type: String,
      enum: ['draft', 'active', 'pending', 'sold', 'withdrawn'],
      default: 'draft',
    },
    { name: 'description', type: String, searchable: true },
    {
      // Multi-file upload field. The framework handles multipart upload,
      // MIME validation, and a per-record file index automatically.
      name: 'photos',
      type: 'File',
      maxFiles: 12,
      allowedMime: ['image/jpeg', 'image/png', 'image/webp'],
    },
    { name: 'listed_at', type: Date, default: Date.now },
  ],
};
```

Create `schema/versions/v1/lead.js`:

```js
module.exports = {
  path: 'lead',
  collection: 'lead',
  fields: [
    { name: 'userId', type: String, required: true },
    { name: 'name', type: String, required: true, searchable: true, searchWeight: 5 },
    { name: 'email', type: String, required: true },
    { name: 'phone', type: String },
    { name: 'interested_in_listing_id', type: String },
    {
      name: 'source',
      type: String,
      enum: ['website', 'referral', 'open_house', 'cold_inbound'],
      default: 'website',
    },
    {
      name: 'stage',
      type: String,
      stateMachine: {
        initial: 'new',
        states: ['new', 'contacted', 'qualified', 'toured', 'closed_won', 'closed_lost'],
        transitions: {
          new: ['contacted', 'closed_lost'],
          contacted: ['qualified', 'closed_lost'],
          qualified: ['toured', 'closed_lost'],
          toured: ['closed_won', 'closed_lost'],
          closed_won: [],
          closed_lost: ['new'],
        },
      },
    },
    { name: 'notes', type: String },
    { name: 'created_at', type: Date, default: Date.now },
  ],
  relations: {
    listing: { belongsTo: 'listing', localKey: 'interested_in_listing_id' },
  },
};
```

Create `schema/versions/v1/viewing.js`:

```js
module.exports = {
  path: 'viewing',
  collection: 'viewing',
  fields: [
    { name: 'userId', type: String, required: true },
    { name: 'leadId', type: String, required: true },
    { name: 'listingId', type: String, required: true },
    { name: 'scheduled_at', type: Date, required: true },
    { name: 'attended', type: Boolean, default: false },
    { name: 'feedback', type: String },
  ],
  relations: {
    lead: { belongsTo: 'lead', localKey: 'leadId' },
    listing: { belongsTo: 'listing', localKey: 'listingId' },
  },
};
```

Hot reload picks all three up. The admin SPA renders the state-machine transition buttons for `lead.stage` automatically — allowed next states come from the schema, the UI doesn't need to know them. See [State machines](/features/state-machines/).

## Seed listings with photos

Register and log in, then add 4–5 listings through the admin UI. For each one, upload 1–2 photos via the multi-file picker. The framework stores them in MongoDB's GridFS by default; for production-scale buckets you'd add [`davepi-plugin-object-storage`](https://github.com/projik/davepi/tree/main/packages/davepi-plugin-object-storage) later.

## Install the Slack plugin

The Slack plugin is a one-way notification channel — totally separate from the `@davepi/agent` interactive bot. Install it now:

```bash
npm install davepi-plugin-slack
```

Add to `package.json`:

```json
{
  "davepi": {
    "plugins": [
      "davepi-plugin-slack"
    ]
  }
}
```

Add to `.env`:

```bash
# Webhook URL from your Slack workspace's Incoming Webhooks app.
SLACK_WEBHOOK_URL=https://hooks.slack.com/services/T.../B.../...

# Which CRUD events trigger a post. Comma-separated patterns
# matched against the type field on each record event.
SLACK_EVENT_PATTERNS=lead.created,lead.transitioned,viewing.created
```

Restart the server. You should see this in the logs:

```
{"plugin":"slack","msg":"plugin loaded","events":["lead.created","lead.transitioned","viewing.created"]}
```

## Watch Slack wake up

Add a new lead through the admin UI. Within about one second, your configured Slack channel posts:

```
🆕 lead.created — name=Maya Chen, email=maya@example.com,
   interested_in=88 Park Slope Ave, stage=new
```

Transition that lead to `contacted` using the admin SPA's state-machine button. The channel posts the transition with `previous` and `next` states.

**You did not write that code.** The plugin subscribed to the framework's `record` event bus and the framework emits an event for every CRUD mutation including state-machine transitions. See [Plugins](/plugins/overview).

## Ask the agent to write an aggregation

Open Claude Code in the project:

> Add an aggregation to lead called `leadsByWeekAndStage` that groups leads by ISO week of `created_at` and by `stage`, sorted chronologically.

Claude adds:

```js
aggregations: [
  {
    name: 'leadsByWeekAndStage',
    description: 'Lead counts grouped by ISO week of created_at and current stage.',
    pipeline: [
      {
        $group: {
          _id: {
            year: { $isoWeekYear: '$created_at' },
            week: { $isoWeek: '$created_at' },
            stage: '$stage',
          },
          count: { $sum: 1 },
        },
      },
      { $sort: { '_id.year': 1, '_id.week': 1, '_id.stage': 1 } },
    ],
    cache: { ttlSeconds: 60 },
  },
],
```

Hot reload exposes it as a new MCP tool: `aggregate_lead_leadsByWeekAndStage`.

## Multi-step lookup without a query

DM the bot:

```
Which leads have been in `qualified` stage for more than 7 days without a scheduled viewing?
```

The agent:
1. Calls `list_lead` with `filter: { stage: 'qualified' }`.
2. For each lead, calls `list_viewing` with the lead id and `scheduled_at: { $gte: <now> }`.
3. Cross-references the two result sets and renders the stale leads as a table.

That's three tool calls choreographed by the model. You didn't write a query for this — the agent assembled it from the available tools.

## Webhooks: ingesting leads from an external portal

Real estate portals push new leads via webhook. Add a public ingest route by asking Claude:

> Add a `POST /api/webhooks/leads` route to the davepi project that accepts an external lead payload `{ name, email, phone, listing_address, source }`, looks up the matching listing by address (or creates a stub if none exists), and creates a lead row with `stage: new`. The route should not require auth (it's called by an external webhook). Expose it as an MCP tool via `expose: true` so the agent can report on recent webhook-ingested leads.

Claude adds the route. Restart the server. Test it:

```bash
curl -X POST http://localhost:5050/api/webhooks/leads \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "Jordan Lee",
    "email": "jordan@example.com",
    "phone": "+15555550100",
    "listing_address": "88 Park Slope Ave",
    "source": "cold_inbound"
  }'
```

The new lead appears in the admin UI with `stage: new`. The Slack plugin fires `lead.created` automatically.

## Lead aggregations: pipeline conversion rate

Ask Claude to add a conversion-rate aggregation:

> Add an aggregation to lead called `conversionBySource` that groups leads by `source` and returns the count at each stage — new, contacted, qualified, toured, closed_won, closed_lost — per source.

Ask the bot once it's loaded:

```
What's our conversion rate from contacted to toured, broken down by lead source?
```

The agent calls `aggregate_lead_conversionBySource`, computes the contacted-to-toured ratio per source in-process, and replies with a ranked summary.
